| Priority | Pts | Ticket | Notes |

| 200 | 2 | **[BB-26] Split ability validation between caster/resource checks and target-cell checks** | Urgent. `CanUseAbility` currently delegates through target-cell discovery, while resource checks live inside `GetCastLegality` / `CanCastAtCell`. Refactor so ability usability answers "can this active unit pay and use this ability at all" without depending on a specific target cell. Target-cell validation should only answer range, LOS, board bounds, and target-profile validity for the selected cell. |
| 195 | 2 | **[BB-27] Add action-level ability validation matching `CanMove`** | Urgent. Add a validator method for resolved `AbilityAction` similar to `BattleActionValidator.CanMove(...)`. `ResolveAbility` should validate the action source, active unit, defeated/placed state, AP/MP availability from `action.ActionPointCost` / `action.MovementPointCost`, and basic action data before resolving. Rework spell selection flow at the same time: selecting a spell should be allowed independently, then only the selected target point should be validated against that spell before submission. |
| 190 | 2 | **[BB-28] Redesign ability target APIs around intent-specific names** | Urgent. Rename or split confusing APIs: `CanUseAbility` for caster/resource readiness, `CanTargetCellWithAbility` or equivalent for target-cell legality, and an action-level validator for submitted `AbilityAction`. Update `PlayerTurnPhase`, `EnemyTurnPhase`, masks, targeting previews, and resolver call sites to use the correct method for each intent. |
| 180 | 2 | **[BB-29] Add tests for ability validation responsibility boundaries** | Urgent. Cover insufficient AP/MP, valid ability with no currently valid target, invalid target cell with enough resources, LOS/range failures, and resolver rejection for an `AbilityAction` whose source can no longer pay. Tests should make it obvious which API is responsible for resource validation vs target-cell validation. |
| 170 | 2 | **[BB-30] Centralize revive lifecycle in `BattleUnitRules`** | Add a `BattleUnitRules.ReviveUnit(...)` path for `ReviveEffect` so revive state restoration, board/placement expectations, and revive feat/event emission are centralized. Keep this ticket focused on revive only. |
